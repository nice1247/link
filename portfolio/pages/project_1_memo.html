Q1. "기존 단일 신청 방식에서 신청자-공모 다중 연결 프로세스로 개편했다"고 하셨는데, 구체적으로 ERD를 어떻게 확장하셨나요? 답변 포인트: 단순 1:1 관계에서 1:N 또는 N:M 관계로의 변화와 정합성 유지를 강조하세요. 모범 답변: "기존에는 신청서 테이블 하나에 신청자와 공모 정보가 종속되어 있어, 한 명의 사용자가 여러 공모에 참여하거나 공모별로 다른 인증 상태를 관리하기 어려웠습니다. 이를 해결하기 위해 '공모 마스터', '신청 정보', '신청자 매핑' 테이블로 분리하고, 공모의 단계별 상태(접수-심사-선정)를 관리하는 히스토리 테이블을 추가하여 데이터의 이력을 추적할 수 있도록 설계했습니다." Q2. 복잡한 공모 로직에서 트랜잭션 안정성을 어떻게 확보했나요? 답변 포인트: eGovFrame(Spring)의 @Transactional 활용과 예외 처리 전략을 설명하세요. 모범 답변: "공모 신청 시 파일 업로드, DB 저장, 알림톡 발송이 한 흐름으로 일어납니다. 이 과정에서 하나라도 실패할 경우 데이터 불일치가 발생할 수 있어 Spring의 선언적 트랜잭션을 적용했습니다. 특히 외부 API(알림톡) 호출은 DB 트랜잭션 완료 후에 수행되도록 이벤트를 분리하거나, 실패 시 재시도 로직을 구현하여 사용자에게 최종적으로 정확한 결과가 전달되도록 했습니다." 2. 보안 및 성능 최적화 관련 질문 Q3. D'Amo 암복호화 솔루션 도입 시 쿼리 레벨 최적화를 수행했다고 하셨는데, 구체적인 사례를 들려주세요. 답변 포인트: 암호화된 컬럼은 인덱스 활용이 어렵다는 점과 이를 해결한 경험을 언급하세요. 모범 답변: "암호화된 컬럼으로 검색할 경우 Full Table Scan이 발생하여 성능이 저하되는 문제가 있었습니다. 이를 해결하기 위해 자주 검색되는 항목은 '일치 검색용 색인 컬럼(Hash 값)'을 별도로 생성하여 인덱스를 태우고, 실제 데이터 노출 시에만 복호화 함수를 호출하도록 쿼리를 최적화했습니다. 이를 통해 대용량 데이터 조회 속도를 기존 대비 약 30% 이상 개선할 수 있었습니다." Q4. 통합 검색(코난테크) 연동 시 백엔드 입장에서 가장 신경 쓴 부분은 무엇인가요? 답변 포인트: DB와 검색 엔진 간의 데이터 동기화를 강조하세요. 모범 답변: "DB에 저장된 기술/제품 정보가 검색 엔진에 실시간 혹은 주기적으로 반영되는 '데이터 동기화'에 가장 신경 썼습니다. 데이터 변경(CUD)이 발생할 때마다 검색 엔진 인덱서에 신호를 보내는 증분 수집(Incremental Indexing) 방식을 설계하여 사용자가 최신 정보를 바로 검색할 수 있도록 구현했습니다." 3. 기술 스택 및 연동 관련 질문 Q5. eGovFrame(전자정부 표준 프레임워크)을 사용하면서 느낀 장점이나 본인만의 활용 노하우가 있나요? 답변 포인트: 표준화된 구조를 통한 유지보수 용이성을 언급하세요. 모범 답변: "공공 프로젝트 특성상 여러 개발자가 협업하는데, eGovFrame의 표준화된 레이어 구조(Controller-Service-DAO) 덕분에 코드 가독성과 유지보수성이 높았습니다. 저는 공통 컴포넌트를 적극 활용하여 반복되는 파일 업로드나 공통 코드 관리 로직을 모듈화함으로써 개발 생산성을 높였습니다." Q6. 다양한 외부 API(알림톡, SMS, 공공데이터)를 통합할 때 발생한 예외 상황을 어떻게 해결했나요? 답변 포인트: Timeout 처리나 API 장애 대응 경험을 설명하세요. 모범 답변: "외부 서비스의 일시적인 장애나 응답 지연이 전체 시스템의 장애로 번지지 않도록 '타임아웃(Timeout)' 설정과 '예외 핸들링'을 철저히 했습니다. 예를 들어 알림톡 발송 API가 응답하지 않을 경우, 즉시 SMS로 전환하여 발송하는 Fallback 로직을 구현하여 메시지 전달의 신뢰성을 확보했습니다." 4. 기타 협업 및 운영 질문 Q7. 고도화 프로젝트는 운영 중인 시스템을 건드리는 작업인데, 기존 데이터 마이그레이션이나 서비스 중단 최소화를 위해 어떤 노력을 했나요? 답변 포인트: 하위 호환성 유지와 철저한 테스트를 강조하세요. 모범 답변: "기존 데이터를 새로운 ERD 구조에 맞게 이관하기 위해 상세 마이그레이션 스크립트를 작성하고 스테이징 환경에서 수차례 검증했습니다. 또한 신규 로직 배포 시 기존 API와의 하위 호환성을 유지하여, 전환 과정에서 발생할 수 있는 사용자 불편을 최소화했습니다." 💡 면접 준비를 위한 추가 팁 ERD 그려보기: 면접관이 화이트보드에 당시 설계한 테이블 구조를 그려보라고 할 수 있습니다. 주요 테이블 간의 관계를 다시 한번 복기하세요. 기존 공모 테이블 유지 하면서 다중공모를 묶어줄 테이블 추가 설계 수치화된 성과 준비: "성능 개선"이라고만 하기보다 "조회 속도 0.5초 단축" 또는 "문의 전화 20% 감소" 등 예상되는 수치를 준비하면 훨씬 신뢰감이 생깁니다. MsSQL 쿼리 경험: MsSQL의 특정 문법(예: Procedure, CTE 등)을 사용했다면 해당 내용을 정리해 두세요. 이 API 가이드는 내부에 구축된 한국어 검색엔진을 REST 방식으로 호출하여, 형태소 분석·인덱싱된 데이터를 기반으로 검색 결과를 반환하는 엔터프라이즈 검색 API 구조를 설명하는 문서다. KONAN Search 4에는 기존의 데이터베이스와의 연동을 위해 DB 크롤러 및 와처가 있습니다. DB 크롤러는 기존 데이터베이스의 레코드를 KONAN Search 4로 수집해 오는 역할을 하고, 와처는 데이터베이스의 변경을 모니터하고 있다가 변경 내용을 실시간으로 검색엔진에 반영합니다.
