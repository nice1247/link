1. 리더십 및 협업 (PL 역할) 질문: "3인 개발팀의 PL로서 코드 리뷰를 진행하셨는데, 리뷰 과정에서 팀원 간의 의견 충돌이 있거나 기술적 수준 차이가 있을 때 어떻게 조율하셨나요?" 의도: 리더로서의 커뮤니케이션 능력과 팀의 코드 품질을 상향 평준화시킨 경험을 확인하려는 질문입니다. 답변 전략: '주관적인 취향'이 아닌 '객관적인 표준'을 강조하세요. 모범 답안: "의견 충돌을 최소화하기 위해 프로젝트 초기에 'Java Convention'과 'API 응답 규격'을 문서화하여 공유했습니다. 리뷰 시에는 개인의 코딩 스타일을 지적하기보다, **'유지보수성'과 '확장성'**이라는 객관적 기준을 제시했습니다. 예를 들어, 특정 로직이 재사용되기 어렵다면 인터페이스화할 것을 권고하는 식이었습니다. 또한 기술적 난도가 높은 부분은 제가 직접 공통 모듈로 개발해 배포함으로써, 팀원들이 핵심 비즈니스 로직에만 집중할 수 있도록 환경을 조성했습니다." 프로젝트 시작시 서비스 흐름, DB 및 API 응답 규격을 다같이 정하고 개인 코딩 스타일은 딱히 터치하지 않음 유사한 비지니스 서비스끼리 묶어 일을 배분 기술적으로 어려운 부분이 있으면 직접 모듈 구성하여 배포하였고 맡은 비지니스 로직에 집중할 수 있도록 환경 조성 2. 기술적 문제 해결 (Redis 세션 관리) 질문: "네이버 앱 등 외부 앱 브라우저에서 세션이 끊기는 현상을 왜 Redis 도입으로 해결하셨나요? 쿠키나 DB 기반 세션 방식과의 차이점은 무엇인가요?" 의도: 특정 기술(Redis)을 선택한 타당성과 인프라에 대한 이해도를 묻는 질문입니다. 답변 전략: '중앙 집중식 세션 관리'와 '성능' 두 가지 측면을 설명하세요. 모범 답안: "외부 앱 브라우저는 각기 다른 인앱 브라우저 특성상 세션 쿠키 처리가 불안정하거나, 로드밸런싱 환경에서 Sticky Session 설정이 미비할 경우 세션 불일치가 발생할 수 있습니다. Tomcat 자체 세션은 메모리 종속적이라 서버 재시작 시 데이터가 날아가지만, Redis를 도입함으로써 서버 상태와 무관하게 세션을 유지하는 Stateless한 구조를 만들었습니다. DB 세션 방식도 고려했으나, 예약 시스템 특성상 빈번한 세션 읽기/쓰기가 발생하므로 I/O 성능이 압도적인 In-memory 기반의 Redis가 가장 적합하다고 판단했습니다." 3. 시스템 설계 (예약 모델 구현) 질문: "시설 대관과 일반 예약은 프로세스가 많이 다를 텐데, 이를 하나의 공통 모듈 기반으로 구현하기 위해 인터페이스를 어떻게 설계하셨나요?" 의도: 추상화 역량과 객체지향 설계(OCP: Open-Closed Principle) 능력을 확인하려는 질문입니다. 답변 전략: 공통점과 차이점을 분리한 구조를 설명하세요. 모범 답안: "두 예약 모델의 공통 분모인 '신청-승인-취소' 과정을 **추상 인터페이스(ReservationService)**로 정의했습니다. 시설 대관처럼 '시간 단위 검증'이 필요한 부분과 공연 예약처럼 '잔여 좌석 검증'이 필요한 부분은 각 서비스 클래스에서 **상속을 통해 상세 로직을 구현(Override)**하도록 설계했습니다. 덕분에 관리자 페이지에서는 예약의 유형을 알 필요 없이 공통 인터페이스를 통해 일괄적인 예약 상태 변경과 통계 처리가 가능해졌고, 새로운 예약 유형이 추가되더라도 기존 코드를 수정하지 않고 확장할 수 있었습니다." 4. 운영 효율성 (관리자 시스템) 질문: "관리자가 코드를 수정하지 않고 예약 로직을 설정하게 만들었다고 하셨는데, 구체적으로 어떤 설정값들을 동적으로 관리하셨나요?" 의도: 시스템의 유연성과 운영자 편의성을 고려한 설계 경험을 묻는 질문입니다. 답변 전략: DB 파라미터화된 설정값들을 예시로 드세요. 모범 답안: "예약의 운영 기간, 최대 신청 인원, 중복 신청 허용 여부, 취소 가능 기간 등을 DB 테이블로 관리했습니다. 특히 복잡한 조건인 '특정 요일 제외'나 '특정 대상(경산 시민 등) 우선 예약' 같은 정책도 관리자 UI에서 JSON 형태나 플래그값으로 설정할 수 있도록 시스템화했습니다. 이를 통해 개발자가 배포를 새로 하지 않아도 재단 운영자가 축제나 전시 성격에 맞춰 즉각적으로 예약 정책을 변경할 수 있도록 하여 운영 효율을 높였습니다." 💡 추가 면접 팁 이 프로젝트는 PL 역할을 수행했다는 점이 가장 큰 강점입니다. 따라서 기술적인 답변 외에도 다음과 같은 태도적 질문에 대비하세요. "일정이 촉박할 때 품질과 마감 기한 중 무엇을 우선하셨나요?" > 둘 다 포기할 수 없는 부분이고 무엇이 더 중요한지 비중을 따져야할 것입니다. 이 프로젝트에선 타협이 힘든 기간에 좀 더 우선순위를 두어 마감기한을 지키되 최소한의 기능은 작동하도록 구현합니다. 이 후 UI/UX 부분을 좀 더 다듬는 방식으로 진행하였습니다. "팀원이 개발한 코드에서 심각한 버그를 발견했을 때 어떻게 피드백하셨나요?" > 버그가 발생한 원인을 왜, 무엇때문에 발생했는지 분석하고 해결하는 방식을 유도하는 방식으로 피드백하였습니다. 3. (추가) 협업 중 의견 충돌 발생 시 질문: "PL로서 기술적 의사결정을 내려야 하는데, 팀원이 본인의 방식이 더 효율적이라며 강하게 반대한다면 어떻게 하시겠습니까?" 답변 핵심: "감정이 아닌 데이터와 근거"를 통한 설득입니다. 모범 답안: > "팀원의 의견이 저와 다르다는 것은 프로젝트를 다각도에서 볼 수 있는 좋은 기회라고 생각합니다. 우선 팀원이 주장하는 방식의 근거(성능, 유지보수성, 개발 속도 등)를 경청하겠습니다. 만약 의견 차이가 좁혀지지 않는다면, **'PoC(기술 검증)'나 '간단한 벤치마킹'**을 제안하겠습니다. 예를 들어 Redis 세션 도입 시에도 '기본 세션으로도 충분하다'는 의견이 있다면, 실제 네이버 인앱 브라우저에서의 재현 테스트 결과를 데이터로 보여주며 설득하는 식입니다. 결국 PL의 역할은 가장 '옳은' 결정을 내리는 것이 아니라, 팀 전체가 '납득할 수 있는' 최선의 결정을 내리는 것이라 믿습니다." > 코드에 정답은 없다고 생각합니다. 팀원이 효율적인 방식이라고 주장한다면 그에 응하는 근거, 이유가 있을 것이고 타당하면 채택하여 진행할 것입니다. 다만 타당하다고 하더라도 기술의 도입으로 인한 유지보수성, 개발속도에 문제를 끼친다면 모든 개발을 완료 후 남는 시간을 통하여 테스트 및 기술교체를 권할 것입니다. 결국 PL의 역할은 가장 '옳은' 결정을 내리는 것이 아니라, 팀 전체가 '납득할 수 있는' 최선의 결정을 내리는 것이라 믿습니다.
